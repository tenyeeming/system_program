Two-Pass 組譯器

OPTAB ｜ SYMTAB ｜ LOCCTR ｜ Object Program 格式

🔧 1. 組譯器是什麼？

組譯器（Assembler）把組合語言原始碼翻譯成機器可執行的目的碼（Object Program）。它需要做這些事：

  • 把 mnemonic 轉換成對應的機器碼（如 LDA → 0x00）
  • 把符號位址（Label）轉換成實際記憶體位址
  • 把資料常數轉換成機器內部表示
  • 產生 Object Program（Header + Text + End records）

為什麼需要兩個 Pass？

⚠️ Forward Reference 問題 在第一行程式碼就可能跳到後面定義的 Label（如 JEQ ENDFIL），而 ENDFIL 到程式後面才定義。 一次掃描沒辦法知道它的位址，所以需要兩個 Pass。

🔄 2. Two-Pass 組譯器架構

Pass 1 建立 SYMTAB → 透過中間檔傳給 Pass 2 → 產生 Object Program
Pass 1（第一遍）Pass 2（第二遍）
主要任務定義符號，建立 SYMTAB產生機器碼，生成 Object Program
輸入原始碼中間檔 + SYMTAB
輸出中間檔 + SYMTABObject Program + Listing
關鍵操作LOCCTR 累加計算每行位址查 OPTAB 和 SYMTAB，組合機器碼

🗂️ 3. 三大資料結構

① OPTAB（Operation Code Table）

儲存每個 mnemonic 對應的 opcode 和格式，使用 Hash Table（查詢 O(1)）。

MnemonicOpcodeFormat
LDA003/4
STA0C3/4
ADD183/4
JSUB483/4
RSUB4C3/4

② SYMTAB（Symbol Table）

Pass 1 掃描到 Label 時，把它的位址存進去；Pass 2 查詢符號位址。也用 Hash Table。

SymbolValue（位址 hex）Flags
COPY1000—
FIRST1000—
CLOOP1003—
ENDFIL1015—
EOF1024—
BUFFER1036—

③ LOCCTR（Location Counter）

追蹤目前組譯到哪個位址，每處理一條指令就加上該指令的長度：

  • 一般指令：+3（3 bytes）
  • WORD：+3
  • RESW n：+3n
  • RESB n：+n
  • BYTE C'xxx'：+字串長度
  • BYTE X'xx'：+1

📄 4. Object Program 格式

Object Program 由三種 Record 組成：

H（Header）→ 多個 T（Text）→ E（End）

完整範例（COPY 程式的 Object Program）

HCOPY  001000 01073       ← Header: 程式名 COPY, 起始 0x1000, 長度 0x1073
T001000 1E 141033482061...  ← Text: 從 0x1000 開始, 30 bytes 目的碼
T00101E 15 0C103A...        ← Text: 從 0x101E 開始, 21 bytes
...
E001000                     ← End: 從 0x1000 開始執行（FIRST 的位址）
🧠 小測驗
Pass 1 結束後，SYMTAB 裡有什麼？Pass 2 靠什麼查出每個 Label 的位址？
查看答案
Pass 1 結束後，SYMTAB 裡有所有 Label 對應的十六進位位址。
Pass 2 靠查 SYMTAB（Hash Table，O(1)）取得每個 Label 的位址，填入指令的 address 欄位。

⚙️ 5. Pass 1 演算法（重點）

Pass 1:
  讀取第一行
  if OPCODE == 'START':
      LOCCTR = 起始位址（operand）
      把 START 行寫入中間檔
      讀下一行
  else:
      LOCCTR = 0

  while OPCODE != 'END':
      if 不是註解:
          if 有 LABEL:
              if LABEL 已在 SYMTAB → 錯誤（重複定義）
              else → 把 (LABEL, LOCCTR) 加入 SYMTAB
          查 OPTAB 找指令長度
          if OPCODE 在 OPTAB → LOCCTR += 3（一般指令）
          elif OPCODE == 'WORD' → LOCCTR += 3
          elif OPCODE == 'RESW' → LOCCTR += 3 * operand
          elif OPCODE == 'RESB' → LOCCTR += operand
          elif OPCODE == 'BYTE' → LOCCTR += 常數長度
          else → 錯誤（無效 opcode）
      把這行（含目前 LOCCTR）寫入中間檔
      讀下一行

  把最後一行（END）寫入中間檔
  儲存程式長度 = LOCCTR - 起始位址

📌 T record 格式速查 T ^ 起始位址(6 hex) ^ 長度(2 hex，十六進位) ^ object code 連在一起

Object Code 補充說明：
ADD TABLE,X 的 object code 為什麼是 18 A0 1B？
SIC 指令 3 bytes：byte0 = opcode（18），bytes 1-2 = [x-bit][15-bit 位址]
TABLE 位址 = 201B，x = 1，所以 bytes 1-2 的最高位強制設成 1：
  位址 201B → 高 7 bits = 20₁₆ 的低 7 位 = 0100000 → 加上最高位 1 → 10100000 = A0₁₆
  低 8 bits 直接填 1B₁₆
→ 最終：18 A0 1B ✓

Object Program： 
H^SUM   ^002000^000BD9
T^002000^15^042018^002018^18A01B^2C2015^382006^0C2BD3^4C0000
T^002018^03^000000
T^002BD6^03^000001
E^002000
說明：
• 程式長度 = 2BD9 - 2000 = BD9₁₆（H record 的長度欄填十六進位）
• T record 第二欄（如 15）是這段資料的 byte 數，也是十六進位：15₁₆ = 21₁₀ bytes
• 第一個 T record 從 2000 到 2014，含 7 條指令 × 3 bytes = 21 bytes = 15₁₆ ✓
• RESW 沒有 object code，所以 T record 遇到 RESW 就必須切斷，下一個 WORD 另開一個 T record

SymbolAddress（hex）說明

TABLE、COUNT、ZERO、TOTAL 等在第 10 行之後定義，Pass 1 掃到第 9 行時還沒有它們的位址，這就是為什麼 Pass 1 必須完整掃完才能解決 Forward Reference。

1-Pass 組譯器的差異：遇到未知的 operand（如 ZERO、TABLE）時，先暫時填 0 並記入 Forward Reference List，等後面遇到定義再回頭修正（Back-Patching）。
